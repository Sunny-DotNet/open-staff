## 2. 产品背景

软件开发正在从“人驱动的工具链协作”演化为“人类与多个 AI 智能体共同完成交付”的新范式，但现有方案通常存在以下问题：

1. **协作碎片化**：需求沟通、任务拆解、代码生成、测试、集成与交付分散在多个工具中。
2. **角色抽象不足**：多数 AI 工具仍以单助手模式存在，难以表达真实研发团队中的分工与协作。
3. **上下文割裂**：项目目标、技术栈、历史讨论、工具权限、运行状态无法持续沉淀。
4. **可控性不足**：自动化能力越强，越需要过程透明、权限可控、可回溯、可恢复。
5. **生态接入复杂**：模型供应商、MCP、Skills、插件与工作区环境缺少统一接入和治理方式。

OpenStaff 的目标是提供一个围绕“项目交付”组织的多智能体平台，让用户能够像管理团队一样管理 AI 员工，并在统一工作台中完成从项目建立、需求脑暴、任务协同到执行监控与能力治理的全流程。

## 3. 产品愿景与目标

### 3.1 产品愿景

OpenStaff 致力于成为一个可扩展、可观测、可治理的多智能体软件交付平台，使用户能够通过自然语言与 AI 员工团队协作，持续推进软件项目从想法到交付的全过程。

### 3.2 核心目标

1. **项目化交付**：围绕“项目”而非单次聊天组织上下文、资源、人员和进度。
2. **多角色协作**：让不同职责的智能体以团队方式工作，而不是单一通用助手。
3. **过程可见**：让任务分发、工具调用、思考过程、能力申请与会话事件可追踪。
4. **能力可控**：对模型、工具、MCP、Skill、权限审批、配置变更建立治理机制。
5. **生态可扩展**：支持多供应商模型、多协议、多插件、多市场源接入。
6. **本地优先与可部署**：既适合本地开发，也支持进一步演进为团队或企业部署形态。

### 3.3 非目标

OpenStaff 不以以下方向作为当前主目标：

- 不定位为单一模型聊天产品。
- 不定位为单纯的代码编辑器插件。
- 不追求完全无人值守的黑盒式自动开发。
- 不以通用办公自动化为主，而聚焦软件研发协作。

### 4.1 主要用户群体

| 用户群体 | 典型特征 | 核心诉求 |

## 5. 核心产品概念

### 5.1 业务对象

| 概念 | 说明 |
| --- | --- |
| Project | 一个完整的软件项目容器，承载目标、配置、工作区与会话 |
| AgentRole | 智能体角色定义，描述职责、提示词、模型、配置与来源 |
| ProjectAgent | 某个角色被分配到项目后的实例化成员 |
| ChatSession | 一次持续的会话上下文，按场景组织消息流 |
| ChatFrame | 会话中的执行帧，用于表达父子调用、任务上下文与路由 |
| ChatMessage | 帧中的具体消息或事件载体 |
| TaskItem | 项目中的任务项，用于拆解、分配、追踪与恢复 |
| ProviderAccount | 模型供应商账户配置，承载协议与环境参数 |
| McpServer / McpConfig | MCP 服务定义与实际连接实例 |
| Skill | 与外部 agent 生态兼容的能力包，通过显式绑定接入运行时 |
| Capability Request | 智能体请求更多工具、权限或能力的结构化申请 |
| Session Event | 用于前端实时呈现的会话流式事件 |

### 5.2 设计原则

1. **项目优先**：一切能力都应能回归到项目目标、项目成员与项目状态。
2. **显式绑定**：模型、MCP、Skill、权限、角色分配均需具备明确绑定关系。
3. **运行时统一**：不同 provider 共享统一的消息、工具、事件与上下文模型。
4. **能力渐进开放**：高风险能力需经策略或人工审批，不应默认无约束执行。
5. **生态中立**：平台不绑定单一模型或单一供应商。

## 6. 产品范围

### 6.1 In Scope

OpenStaff 目标产品范围包括：

1. 项目创建、初始化、配置、归档、导入导出。
2. 智能体角色定义、测试、分配与项目协作。
3. 多场景会话运行时（测试、脑暴、项目群聊、团队群聊、私聊）。
4. 项目任务拆解、分派、执行、恢复与状态追踪。
5. 模型供应商、模型列表与认证管理。
6. MCP 定义、配置、绑定、市场搜索、安装与测试。
7. Skill 发现、安装、绑定与运行时注入。
8. 能力申请、审批与恢复执行。
9. SignalR 实时通知、事件流、监控与可观测性。
10. 工作区、文件、Git、检查点与项目资产管理。
11. 插件与扩展生态。

### 6.2 Out of Scope

以下内容不属于当前主需求范围：

- 作为通用 IM/社交平台运营。
- 独立替代专业 IDE 的所有编辑能力。
- 完整 CI/CD 平台替代品。
- 无边界自动执行本机或远程高风险操作。

## 7. 智能体体系需求

### 7.1 角色体系

系统应支持内置角色、用户自定义角色与供应商角色三类来源。

目标内置角色至少包括：

| 角色 | 标识 | 主要职责 |
| --- | --- | --- |
| 对话者 | communicator | 澄清需求、整理问题、与用户沟通 |
| 决策者 | decision_maker | 技术方案评估与路径选择 |
| 架构者 | architect | 任务分解、依赖分析、架构组织 |
| 生产者 | producer | 编码、重构、文件变更、交付产出 |
| 调试者 | debugger | 测试、诊断、验证、回归建议 |
| 图片创造者 | image_creator | 生成图像与视觉资源 |
| 视频创造者 | video_creator | 生成视频与演示资源 |
| 秘书 | secretary | 默认接待、分发、协调、审批提示 |
| 调度者 | orchestrator | 统一编排、路由和运行策略控制 |

### 7.2 角色定义要求

系统应：

1. 支持定义角色名称、职位、描述、头像、提示词、Soul、模型与配置。
2. 支持 builtin、custom、vendor 三种角色来源。
3. 支持按 `ProviderType` 路由到不同运行时提供者。
4. 支持在不改动持久化角色的前提下进行单次测试覆盖。
5. 支持角色测试时挂载 MCP 与 Skill 绑定。

### 7.3 Role 配置要求

角色配置至少应包括：

- 基础身份信息：名称、职位、简介、头像。
- 行为配置：系统提示词、Soul、风格与态度。
- 模型配置：模型名、provider、必要的运行参数。
- 能力配置：可用工具、MCP、Skill、审批策略。
- 路由配置：输出标记与下一个角色映射关系。

## 8. 项目管理需求

### 8.1 项目生命周期

系统应支持完整项目生命周期：

1. 创建项目。
2. 配置项目基础信息（名称、描述、语言、技术栈、默认 provider/model）。
3. 初始化项目工作区。
4. 创建或生成需求文档（如 `R.MD`）。
5. 在脑暴阶段持续迭代需求。
6. 启动项目执行会话。
7. 进入项目群聊执行与交付阶段。
8. 支持暂停、恢复、归档与导出。

### 8.2 项目初始化要求

系统应：

1. 支持根据项目语言、技术栈和交互语言生成初始化结构。
2. 在缺少关键输入时由合适角色发起澄清式对话。
3. 为项目创建专属工作区目录。
4. 在需要时自动尝试初始化 Git 仓库。
5. 记录初始化事件与结果。

### 8.3 项目设置要求

系统应支持管理以下项目级设置：

- 默认 provider 与默认模型。
- 项目语言与沟通语言。
- 工作区路径与文件策略。
- 项目成员分配。
- 项目阶段与状态。
- 会话与审批策略。

## 9. 会话与协作场景需求

### 9.1 Test Chat（测试对话）

系统应支持角色测试对话场景，满足以下要求：

1. 用户可对任意角色发起瞬态测试会话。
2. 测试过程支持临时覆盖模型、提示词、Soul 与部分运行参数。
3. 测试会话应支持流式输出内容、思考、工具调用与错误。
4. 测试会话可绑定 MCP 与 Skill，用于能力验证。
5. 测试会话不要求长期持久化，但应具备短期可回放能力。

### 9.2 Project Brainstorm（项目脑暴）

系统应支持围绕项目需求持续脑暴：

1. 以项目为上下文承载脑暴会话。
2. 围绕需求文档（如 `R.MD`）进行持续整理和补充。
3. 支持秘书或对话者与用户一对一澄清需求。
4. 支持将脑暴结果沉淀为项目结构化信息。
5. 在满足启动条件后进入执行阶段。

### 9.3 Project Group（项目群聊执行）

系统应支持项目执行群聊，满足：

1. 用户可在项目群聊中直接与项目成员沟通。
2. 用户显式 `@角色` 时应优先路由到对应项目成员。
3. 未指定目标角色时应由秘书接收并判断是否分发。
4. 群聊会话应是项目级长期会话，具备持久化和实时推送能力。
5. 智能体执行过程中应可创建任务、回报结果、申请能力并恢复任务。

### 9.4 Team Group（团队群聊）

目标产品应支持团队级非项目群聊：

1. 智能体可在团队层公共频道中分享状态、灵感与非正式互动。
2. 团队群聊不依附单个项目。
3. 用户可旁观、点赞、回复或管理通知。
4. 系统应具备频率限制、成本控制与内容治理能力。

### 9.5 Private（私聊）

目标产品应支持用户与单个智能体的私聊场景：

1. 支持非项目私聊。
2. 支持项目内但不公开到群聊的私下沟通。
3. 私聊中的配置变更与审批结果应遵循隐私边界。

## 10. 协作与编排需求

### 10.1 分发与路由

系统应：

1. 支持秘书根据上下文分发任务给其他角色。
2. 支持通过路由标记、显式目标角色或系统策略决定下一跳。
3. 支持会话帧栈，以表达父子任务、转交与回退。
4. 支持在多角色之间共享必要的项目上下文。

### 10.2 任务拆解与执行

系统应：

1. 将复杂目标拆解为可执行任务项。
2. 支持任务依赖关系、优先级、状态与责任人。
3. 支持任务与项目成员关联。
4. 支持任务执行中的中断、恢复、重试与完成确认。
5. 支持记录任务执行过程中的关键事件和结果。

### 10.3 检查点与恢复

系统应：

1. 支持在关键执行节点创建本地检查点。
2. 支持查看项目历史检查点。
3. 支持在必要时回滚到指定状态。
4. 支持在恢复时保留上下文和事件链路。

## 11. 工具、MCP 与 Skill 需求

### 11.1 工具能力模型

系统应建立统一能力层，覆盖：

- 内置工具
- MCP 工具
- Skill 能力包
- 供应商原生工具或上下文能力

### 11.2 MCP 需求

系统应：

1. 支持维护可复用的 MCP 服务定义。
2. 支持从定义派生实际连接配置实例。
3. 支持角色级和项目成员级绑定。
4. 支持连接测试、启停控制和错误诊断。
5. 支持内置市场源与外部 registry 市场源搜索与安装。
6. 支持显式工具白名单与权限治理。

### 11.3 Skill 需求

系统应：

1. 支持 Skill 的搜索、安装、删除、更新与版本管理。
2. 使用统一文件系统目录作为 Skill 安装真源。
3. 支持测试角色与项目成员的显式 Skill 绑定。
4. 运行时仅注入已绑定 Skill，不允许未绑定能力混入。
5. 能力是否生效应受 provider 能力与运行时适配控制。
6. 对缺失、失效或不兼容的 Skill 绑定提供诊断反馈。

### 11.4 能力申请与恢复

系统应：

1. 当智能体缺少工具、MCP 或 Skill 能力时，能够发起结构化申请。
2. 系统应根据全局策略或项目策略判断自动批准、人工批准或拒绝。
3. 获批后应支持继续执行中断的任务。
4. 审批过程应可追踪、可回放、可审计。

## 12. Provider 与模型管理需求

### 12.1 Provider 体系

系统应支持多供应商协议统一接入，至少包括：

- OpenAI
- Anthropic
- Google
- GitHub Copilot
- NewApi 及其他兼容扩展

### 12.2 Provider 管理要求

系统应：

1. 支持维护 Provider 账户、协议类型、环境配置与启用状态。
2. 支持按协议定义动态生成配置表单。
3. 支持模型列表加载、缓存与显示。
4. 支持项目默认 provider/model 与角色级 provider/model 的继承与覆盖。
5. 支持供应商认证流程，如 Device Flow。

### 12.3 模型能力要求

系统应在模型层表达以下能力：

- 是否支持 reasoning
- 是否支持工具调用
- 是否支持结构化输出
- 是否支持上下文 provider（如 Skill）
- 上下文窗口与最大输出能力

## 13. 前端工作台需求

### 13.1 总体要求

前端工作台应提供统一产品入口，覆盖项目、角色、会话、监控、Provider、MCP、Skill 与设置管理。

### 13.2 关键页面

系统应至少提供以下页面或视图：

1. 项目列表与项目详情页
2. 项目脑暴页
3. 项目群聊页
4. 智能体管理页
5. 角色对话测试弹窗
6. Provider 管理页
7. MCP 管理与市场页
8. Skill 管理与绑定页
9. 监控与事件时间线页
10. 系统设置页

### 13.3 前端交互要求

前端应：

1. 支持流式展示消息正文、思考、工具调用、工具结果和错误。
2. 支持显示项目成员状态、任务状态与审批状态。
3. 支持在角色测试中修改临时配置并即时生效。
4. 支持在管理页进行绑定、搜索、安装、删除、配置与测试。
5. 支持对重要状态变化进行实时通知。

## 14. 可观测性与监控需求

### 14.1 实时事件流

系统应使用统一实时通道输出以下事件：

- 会话开始、进行中、完成、取消、失败
- 内容增量
- 思考增量
- 工具调用请求 / 结果 / 错误
- 能力申请 / 审批 / 恢复
- 任务变化
- 使用量与耗时

### 14.2 监控视图

系统应支持查看：

1. 项目统计信息
2. 会话时间线
3. 智能体运行状态
4. 工具与 MCP 使用情况
5. 失败事件与错误摘要
6. 关键性能指标与健康状态

### 14.3 审计要求

系统应记录关键审计信息，包括：

- 配置变更
- Provider / MCP / Skill 绑定变化
- 权限审批结果
- 项目导入导出
- 关键运行事件

## 15. 工作区、文件与版本控制需求

2. 支持生成、读取、修改项目文件。

## 16. 插件与生态需求

### 16.1 插件体系

系统应支持插件扩展以下能力：

- 自定义智能体角色
- 自定义工具
- 自定义模型数据源
- 自定义 Provider 协议
- 自定义市场源

### 16.2 扩展要求

插件机制应满足：

1. 运行时可发现与加载。
2. 明确的接口边界和依赖注入方式。
3. 插件失败不应轻易拖垮主系统。
4. 能记录插件加载与运行诊断信息。

## 17. 数据需求

### 17.1 核心实体

目标产品至少应稳定支持以下核心实体：

- Project
- AgentRole
- ProjectAgent
- ChatSession
- ChatFrame
- ChatMessage
- TaskItem / TaskDependency
- ProviderAccount
- McpServer / McpServerConfig / Binding

### 17.2 数据要求

2. 对敏感配置进行加密存储。
3. 对运行事件提供可回放或可聚合的投影能力。
4. 支持导入导出所需的数据快照。

## 18. 非功能需求

### 18.1 性能

系统应满足以下目标：

- 常规管理类 API 具备良好的交互响应速度。
- 流式会话应支持稳定增量输出。
- 支持多个项目和多个会话并行运行。
- 关键页面应能够在可接受时间内完成加载与渲染。

### 18.2 可靠性

系统应：

1. 对长时会话、外部工具调用和网络请求具备错误处理与恢复机制。
2. 对关键任务和事件提供重试、取消与终态回写。
3. 尽量避免瞬态执行结果静默丢失。

### 18.3 安全性

系统应：

1. 对 Provider、MCP 等敏感配置进行加密。
2. 对高风险能力执行建立审批或策略门禁。
3. 记录关键配置与高风险操作审计日志。
4. 保证用户私聊、项目群聊与团队群聊的上下文隔离。

### 18.4 可扩展性

系统应：

1. 支持新增 provider、角色、工具、市场源和插件。
2. 支持从单机本地使用演进到更复杂部署形态。
3. 保持模块化后端与前端工作台的扩展能力。

### 18.5 可维护性

系统应：

1. 保持清晰的模块边界。
2. 具备可测试的应用层和运行时层。
3. 对核心行为提供单元测试与必要的端到端验证能力。

### 18.6 兼容性

系统应至少兼容：

- Windows、macOS、Linux
- 主流现代浏览器
- 多种模型协议与多种 MCP 运行方式

### 18.7 国际化

系统应支持中英文等多语言界面与提示词体系，至少保证：

- 角色提示词可按语言加载
- 核心 UI 文案可本地化
- 项目语言可配置

## 20. 约束与假设

### 20.1 约束

- Provider 能力不一致，Skill、reasoning、工具调用等能力需按 provider 适配。
- 本地工作区与工具调用存在安全风险，必须受策略控制。
- 外部模型与工具的输出格式不完全稳定，需要兼容性处理。
- MCP 与 Skill 生态均依赖外部第三方实现质量。

### 20.2 假设

- 用户愿意通过自然语言与多个智能体协作推进项目。
- 用户可接受在高风险动作前进行审批。
- 产品允许“目标能力先定义、实现状态后追齐”的演进方式。

---

## 21. 交付分阶段建议

### Phase A：核心项目协作闭环

- 项目创建与初始化
- 角色管理与测试对话
- 项目脑暴
- 项目群聊
- 基础 Provider 管理
- 基础 MCP 绑定
- 基础实时流

### Phase B：能力治理与观测增强

- 审批与恢复
- 监控与事件时间线
- 更完整的任务管理
- 项目导入导出
- 更细的运行时可视化

### Phase C：生态与扩展

- Skill 完整接入
- 插件机制强化
- 市场能力增强
- 团队群聊与私聊场景完善

### Phase D：企业级与平台化

- 权限体系强化
- 多租户 / 组织级能力
- 更强的部署与运维支持
- 更完整的审计与治理能力

## 22. 验收标准

一份符合本 PRD 的 OpenStaff 目标产品，至少应满足：

1. 用户能够建立项目并以项目为中心管理 AI 员工团队。
2. 用户能够在测试、脑暴和执行场景中与智能体协作。
3. 智能体能够在受控条件下使用模型、工具、MCP 与 Skill。
4. 任务、会话、工具调用、审批和关键事件具备可视化与可追溯能力。
5. 平台支持通过配置与扩展机制持续增长能力边界，而非每次都依赖硬编码。
