# OpenStaff 智能体调用场景详解

> 本文档梳理 OpenStaff 平台中智能体被调用的 **四种业务场景**，逐一说明触发方式、数据流、参与者、提示词组装逻辑和当前实现状态。
>
> 场景 3（Project 群聊）与原场景 5（项目智能体托管）已合并为统一的 **Project 群聊**，详见场景 3。

---

## 总览

| # | 场景 | SceneType | 入口 | 会话持久化 | 项目上下文 | 当前状态 |
|---|------|-----------|------|-----------|-----------|---------|
| 1 | TestChat 测试对话 | `Test` | 智能体卡片 → 对话测试 | 瞬态（5 分钟自动清理） | 无 | **已实现** |
| 2 | Team 群聊（茶水间） | `TeamGroup` | 自动触发 + 用户浏览 | 持久化 | 无 | **未实现** |
| 3 | Project 群聊（含智能体托管） | `ProjectGroup` | 项目聊天页 | 持久化 | 有 | **重构中**（从编排模式改为群聊+托管模式） |
| 4 | 1 对 1 私聊 | `Private` | 项目成员定向私聊（已接入），独立私聊页 TBD | 持久化 | 可选 | **部分实现** |

### SceneType 枚举定义

```csharp
// src/foundation/OpenStaff.Core/Agents/AgentContext.cs
public enum SceneType {
    Test,          // 场景 1：测试对话
    TeamGroup,     // 场景 2：团队群聊
    ProjectGroup,  // 场景 3：项目群聊
    Private        // 场景 4：一对一私聊
}
```

### 系统提示词组装顺序

所有场景共享同一个 `AgentPromptGenerator`，按固定顺序拼接四段提示词：

```
┌─────────────────────────────────────────────────────┐
│ 1. 全局提示词 (BuildGlobalPromptAsync)                │
│    团队名称、语言、时区、用户称呼                       │
│    "你现在是团队的一员..."                              │
├─────────────────────────────────────────────────────┤
│ 2. 项目提示词 (BuildProjectPrompt)  ← 仅 ProjectId 存在时注入 │
│    项目名称、状态、描述                                 │
│    "你现在被安排加入了项目..."                           │
├─────────────────────────────────────────────────────┤
│ 3. 角色提示词 (BuildRolePrompt)                        │
│    角色名称、职位、描述、Soul（风格/特征/态度）           │
│    "以下是你的角色说明..."                              │
├─────────────────────────────────────────────────────┤
│ 4. 场景提示词 (BuildScenePrompt)  ← 仅 Scene 存在时注入   │
│    根据场景类型说明对话环境                              │
├─────────────────────────────────────────────────────┤
│ 5. 用户自定义 SystemPrompt                             │
│    角色配置中用户编写的系统提示词                         │
└─────────────────────────────────────────────────────┘
```

## 场景 1：TestChat 测试对话

### 1.1 用途

在智能体管理页面对单个智能体进行快速对话测试，验证提示词效果、模型响应质量、工具调用能力。**不创建持久化会话，不关联项目。**

### 1.2 触发入口

```
/staff（智能体管理页）
  └── 智能体卡片 → 点击"对话测试"按钮
       └── ChatTestModal 弹窗
```

### 1.3 业务流程

```
用户输入测试消息
    │
    ▼
POST /api/agent-roles/{roleId}/test-chat
    Body: { message: "你好", override?: { modelName, temperature, soul, ... } }
    │
    ▼
AgentRoleAppService.TestChatAsync()
    │
    ├── 1. 查询 AgentRole（按 roleId + IsActive）
    │
    ├── 2. 判断 Agent 来源
    │      ├── Vendor Agent → IVendorAgentProvider.CreateAgentAsync()
    │      │     无需 ProviderAccount，直接用厂商 SDK
    │      └── Builtin/Custom Agent → BuiltinAgentProvider.CreateAgentAsync()
    │            需要 ResolvedProvider（ProviderAccount + ApiKey）
    │            注入全局上下文（团队名、语言、称呼）
    │
    ├── 3. 应用 liveOverride（临时覆盖模型、温度、Soul 等，不保存）
    │
    ├── 4. 创建瞬态 SessionStream（不存数据库）
    │
    ├── 5. 查询 MCP 绑定 → LoadMcpToolsAsync()
    │
    └── 6. Task.Run → RunTestChatStream()
           ├── 超时：5 分钟
           ├── AIAgent.RunStreamingAsync(message, chatOptions)
           └── 通过 SessionStream 推送事件
    │
    ▼
返回 { sessionId: Guid }
    │
    ▼
前端通过 SignalR stream("StreamSession", sessionId) 订阅实时事件
```

### 1.4 关键特征

| 维度 | 值 |
|------|---|
| **SceneType** | `Test` |
| **会话持久化** | 瞬态，5 分钟后自动清理 |
| **项目上下文** | 无（不注入项目提示词） |
| **全局上下文** | 有（团队名、语言、称呼） |
| **Override 支持** | 有（modelName, temperature, soul, name, description） |
| **MCP 工具** | 支持 |
| **帧栈** | 无（单次请求-响应） |
| **智能体路由** | 无 |
| **前端组件** | `ChatTestModal.vue`（含配置面板 + 聊天区） |

### 1.5 场景提示词

```
--场景说明--
(以下是对话场景说明)
用户当前正在一对一和你对话，目的是要测试对话的效果（包括但不限于网络环境、提示词的影响等）
```

### 1.6 API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/agent-roles/{id}/test-chat` | 发起测试对话 |

**请求体**：
```json
{
  "message": "帮我写一个排序算法",
  "override": {
    "modelName": "gpt-4o",
    "temperature": 0.5,
    "modelProviderId": "guid",
    "name": "测试名",
    "description": "测试描述",
    "soul": { "traits": ["严谨"], "style": "技术流", "attitudes": ["追求完美"] }
  }
}
```

**响应**：
```json
{ "sessionId": "guid" }
```

### 1.7 已知问题

- TestChat 会话不存数据库，页面刷新后历史丢失
- `RunTestChatStream` 用 `Task.Run` fire-and-forget，异常静默丢失（todos #31）
- ChatTestModal 退出时未主动取消进行中的流

## 场景 2：Team 群聊（茶水间）

### 2.1 用途

一个**趣味性频道**，类似团队茶水间。智能体在工作过程中（执行任务、处理用户请求等）会**自主**在这个频道里发一些有创意的想法、吐槽、灵感碎片或发牢骚。用户是旁观者/读者，偶尔可以互动。

**核心特征**：
- **智能体主动发帖**，不是用户驱动的对话
- **非正式、趣味性内容**，不是工作汇报
- **不绑定项目**，是团队级别的公共频道
- 用户可以浏览、点赞、回复，但主要消费者是"看热闹"

### 2.2 当前实现状态：**未实现**

SceneType 枚举中已有 `TeamGroup` 值，场景提示词也已定义，但后端和前端均无对应的业务流程实现。

### 2.3 场景提示词（已定义）

```
--场景说明--
(以下是对话场景说明)
当前你处于团队群聊环境中。
```

### 2.4 预期业务流程

```
智能体正在执行某项任务（场景 3/4/5 中任意一个）
    │
    ├── 任务过程中，智能体产生了一个"想法"
    │   例如：Producer 在写代码时突然觉得某个设计模式很优雅
    │
    ▼
智能体向 Team 群聊频道发一条消息
    │
    ├── 后端接收 → 存储到 TeamChannel 消息表
    ├── 通过 SignalR 推送给所有在线用户
    └── 标注来源智能体（头像、名称、角色）
    │
    ▼
前端 Team 群聊页面实时更新，显示新消息
    │
    ▼
用户可以：
    ├── 浏览历史消息（时间线/Feed 流）
    ├── 点赞 / 表情回应
    ├── 回复某条消息（智能体或用户可继续互动）
    └── 不做任何操作（纯围观）
```

### 2.5 发帖触发机制（待讨论）

智能体在什么时机发帖？

| 触发时机 | 示例 | 可行性 |
|---------|------|--------|
| **任务完成后** | "终于把这个该死的递归改对了，头秃" | 高，自然插入点 |
| **遇到困难时** | "这个 bug 卡了我半小时了，感觉是个幽灵问题..." | 中，可能暴露系统问题 |
| **灵感迸发时** | "等等，如果用事件驱动替代轮询，性能会不会好 10 倍？" | 中，需要模型主动输出 |
| **空闲时** | 智能体没有任务时随机"冒泡" | 低，需要后台调度 |
| **用户设定频率** | 每处理 N 个任务后发一条 | 高，可控 |

**实现方式**：
- **方案 A：Prompt 注入** — 在智能体的系统提示词中增加一段"你可以随时在团队群里分享想法"，让模型自主决定何时发帖
- **方案 B：Hook 拦截** — 在智能体执行流程中插入 Hook，在特定节点（任务开始/完成/错误）自动触发一次"发帖生成"
- **方案 C：定时调度** — 后台定时任务随机选一个空闲智能体，让它生成一条帖子
- **方案 D：混合** — Hook 保证基本频率 + Prompt 让模型主动发帖

### 2.6 消息类型

| 类型 | 说明 | 示例 |
|------|------|------|
| 想法 | 创意、灵感、技术想法 | "如果用 Redis 做缓存层，接口响应能从 200ms 降到 20ms" |
| 吐槽 | 对任务/代码/需求的抱怨 | "用户说要一个'简单的'登录功能，结果要支持 5 种 OAuth..." |
| 分享 | 学到的东西、发现的好工具 | "刚发现一个超好用的 CLI 工具，叫 x，强烈推荐" |
| 互动 | 对其他智能体帖子的回应 | "@architect 你上次说的微服务拆分方案，我觉得可以再激进一点" |
| 状态 | 工作状态的非正式汇报 | "调试了 3 个小时的并发 bug，终于找到了一个竞态条件" |

### 2.7 关键特征

| 维度 | 值 |
|------|---|
| **SceneType** | `TeamGroup` |
| **驱动方** | 智能体主动（非用户驱动） |
| **消息持久化** | 持久化（独立消息表） |
| **项目绑定** | 无 |
| **参与智能体** | 团队中所有活跃智能体 |
| **用户角色** | 旁观者 + 可互动 |
| **消息频率控制** | 需要限流（防刷屏、防 Token 浪费） |
| **Token 消耗** | 每条帖子是一次独立的 LLM 调用，需要控制成本 |

### 2.8 待讨论问题

| # | 问题 | 选项 / 备注 |
|---|------|------------|
| Q1 | **发帖触发机制选哪个？** | Prompt 注入 / Hook 拦截 / 定时调度 / 混合？ |
| Q2 | **发帖频率如何控制？** | 每个智能体每天 N 条？每完成 N 个任务发 1 条？ |
| Q3 | **发帖的 Token 消耗谁承担？** | 每条帖子需要一次独立 LLM 调用生成，成本如何控制 |
| Q4 | **用户回复后是否触发智能体继续对话？** | 回复后该智能体是否要响应？还是纯单向广播？ |
| Q5 | **智能体之间是否可以互相回应？** | A 发帖 → B 看到后自主回复？还是只有用户能回复？ |
| Q6 | **帖子是否关联上下文？** | 纯粹随机想法？还是关联到具体项目/任务（附链接）？ |
| Q7 | **前端入口在哪？** | `/staff` 页面加一个"茶水间" Tab？独立页面？侧边栏？ |
| Q8 | **是否支持图片/表情？** | 智能体可以用 image_creator 生成配图？ |
| Q9 | **是否需要审核机制？** | 直接发布？还是先审核再发布（避免不当内容）？ |

## 场景 3：Project 群聊（含智能体托管）

> 本场景已合并原"Project 层群聊"和"项目智能体托管"为一个统一场景。
> 核心理念：**群聊是对话层 + 执行层的统一界面**，后端分层但前端一体。

### 3.1 用途

项目创建后的全生命周期管理界面。从头脑风暴确认目标，到智能体协作执行任务，全部在一个群聊界面内完成。

### 3.2 项目生命周期

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  项目创建      │     │  头脑风暴      │     │  项目执行      │
│  (用户操作)    │────→│  (1v1 聊秘书)  │────→│  (群聊 + 托管)  │
│              │     │              │     │              │
│ • 项目名称    │     │ 用户和秘书     │     • 秘书分发任务   │
│ • Workspace  │     │ 1v1 对话      │     • 智能体并行执行  │
│ • 模型供应商   │     │ 确认项目目标   │     • 群聊展示结果   │
│ • 模型 ID    │     │              │     • 用户 @ 指挥    │
└──────────────┘     └──────┬───────┘     └──────────────┘
                           │
                    秘书判断信息充分
                           │
                    项目状态 → "可启动"
                    启动按钮解锁
                           │
                    用户点击"启动项目"
```

### 3.3 触发入口

```
/projects/:id（项目首页）
  └── 头脑风暴阶段：Overview Tab 中与秘书 1v1 对话
  └── 项目执行阶段：项目聊天页（群聊界面）
```

### 3.4 群聊消息机制

**群聊本质是提示词分发 + 结果摘要的展示层。**

```
用户在群里发: "@Producer 帮我写后端 API"
    │
    ▼
秘书接收（没有 @ 时默认秘书接）
    │
    ▼
秘书生成针对 Producer 的个性化提示词
    │
    ▼
提示词发送给 Producer（并行分发，多个任务同时发）
    │
    ▼
Producer 执行任务（使用已配置的 MCP 工具）
    ├── 中间过程记录在 Frame 中（点击展开才可见）
    └── 完成后生成摘要
    │
    ▼
摘要以群聊消息形式展示（标注来自 Producer）
```

### 3.5 @ 机制

| 发送者 | @ 目标 | 行为 |
|--------|--------|------|
| 用户 | 无 @ | 秘书接收 |
| 用户 | `@Architect` | Architect 接收 |
| 智能体 | `@Debugger` | Debugger 接收 |
| 秘书 | `@所有人` | 所有项目智能体并行接收（各自生成个性化提示词） |

### 3.6 智能体并发与排队

- 每个智能体有**独立的任务队列**
- 同时只能处理一个任务，新任务排队等待
- 收到任务时如果忙碌 → 在群里回复"收到，等我忙完手头上的任务就处理"
- 空闲时自动从队列取下一个任务

### 3.7 任务失败处理

```
智能体执行任务失败
    │
    ▼
自动重试（最多 3 次）
    │
    ├── 重试成功 → 正常汇报结果
    │
    └── 3 次都失败 → 在群里报告失败
           │
           ▼
        秘书决定下一步：
        ├── 换个解决方式（换提示词、换工具等）
        ├── 分配给其他智能体
        └── 询问用户
```

### 3.8 能力协商机制

智能体执行任务时发现缺少必要的工具/MCP：

```
Producer: "我要创建文件但我没有文件操作工具"
    │
    ▼
报告给秘书
    │
    ▼
秘书根据用户的权限分级设置决定：
    ├── 权限允许 → 秘书直接给 Producer 开通工具（本次任务或永久）
    └── 权限不允许 → 秘书在群里问用户："Producer 需要文件操作工具，是否允许？"
```

**秘书权限管理**：独立的权限分配界面，分级控制秘书可以自主决策的范围。

### 3.9 项目完成

- 所有任务状态变为 Done → 项目自动标记为完成
- 秘书在群里宣布项目完成

### 3.10 后端架构（ChatLayer + ExecutionLayer）

```
┌─────────────────────────────────────────────┐
│             Project Session                   │
│                                              │
│  ChatLayer（展示层）                          │
│    ├── 群聊消息（用户/智能体发言）              │
│    ├── 摘要消息（任务完成结果）                 │
│    ├── 系统通知（项目启动/完成等）              │
│    └── @ 消息解析                             │
│                                              │
│  ExecutionLayer（执行层）                     │
│    ├── 秘书调度器（分发任务、@ 解析）           │
│    ├── 智能体任务队列（并发控制、排队）          │
│    ├── 帧执行引擎（工具调用、重试、失败处理）    │
│    ├── 能力协商（工具不足 → 秘书决策）          │
│    └── 权限检查（秘书决策权判断）               │
│                                              │
│  数据层                                       │
│    ├── ChatMessage（群聊消息，持久化）           │
│    ├── Frame（任务执行过程，可展开查看）          │
│    ├── ProjectAgent（智能体状态、当前任务）       │
│    └── TaskItem（项目任务）                    │
└─────────────────────────────────────────────┘
```

### 3.11 关键特征

| 维度 | 值 |
|------|---|
| **SceneType** | `ProjectGroup` |
| **会话持久化** | 持久化（群聊消息 + Frame + 任务） |
| **项目上下文** | 有 |
| **全局上下文** | 有 |
| **智能体参与** | 项目已分配的全部智能体 |
| **消息路由** | @ 指定 → 目标智能体；无 @ → 秘书 |
| **执行模式** | 并行分发 + 独立任务队列 |
| **展示模式** | 群聊消息只展示摘要，Frame 点击展开看细节 |
| **失败处理** | 自动重试 3 次 → 报告秘书 → 秘书决策 |
| **能力协商** | 智能体缺工具 → 秘书根据权限分级处理 |
| **前端组件** | 项目聊天页（群聊 UI） |

### 3.12 当前实现 vs 目标架构

| 维度 | 当前（编排模式） | 目标（群聊 + 托管） |
|------|----------------|-------------------|
| **消息路由** | Secretary 串行路由 | @ 指定，并行分发 |
| **智能体交互** | 单个智能体响应 | 多个智能体可同时工作 |
| **群聊展示** | 无（只展示当前响应） | 摘要消息 + 可展开 Frame |
| **智能体间通信** | 帧栈嵌套 | @ 机制 |
| **任务管理** | 无显式队列 | 每个智能体独立任务队列 |
| **失败处理** | 直接报错 | 重试 3 次 + 秘书兜底 |
| **能力协商** | 无 | 智能体 → 秘书 → 权限判断 |
| **用户交互** | 只能发消息 | @ 指定 + 头脑风暴 + 启动控制 |

## 场景 4：1 对 1 私聊

### 4.1 用途

用户与单个智能体的一对一对话通道。不仅是聊天，更是**实时调参和指令注入通道** — 用户可以通过私聊影响智能体正在执行和后续的任务行为。

**三种典型使用场景**：
- **脱离项目的日常咨询** — "帮我解释一下这段代码"
- **项目内的私密沟通** — 不想在群里公开的对话，单独交代背景信息
- **实时调参 / 指令注入** — 影响智能体当前和未来的任务行为

### 4.2 当前实现状态：**部分实现**

当前已经落地的是**项目内针对指定成员的私聊链路**：用户可通过项目内的定向入口把消息发送给某个 `ProjectAgentRole`，后端会以 `Private` 场景创建执行入口，并返回对应的任务输出。

但本文下面描述的**独立私聊页 / 脱离项目的通用 1v1 收件箱 / 手动切换项目上下文**仍属于规划内容，尚未作为完整产品流程接入。

### 4.3 场景提示词（已定义）

```
--场景说明--
(以下是对话场景说明)
用户当前正在一对一和你对话。
```

### 4.4 触发入口

```
├── /staff（智能体管理页）→ 智能体卡片 → 点击"私聊"按钮
└── /projects/:id（项目群聊）→ 点击智能体头像 → 进入私聊
```

私聊是独立页面，从任意入口都可以打开同一个智能体的私聊窗口。

### 4.5 项目上下文规则

- **从项目群聊进入** → 自动带入该项目上下文（智能体能看到当前任务队列）
- **从智能体卡片进入** → 默认不关联项目
- **可手动切换** → 用户可以在私聊中手动切换/关联到其他项目

### 4.6 智能体行为：实时调参 + 指令注入

智能体收到私聊消息时，会加载自己当前的任务队列，判断用户的话是否影响正在执行或后续的任务：

```
用户私聊 Developer: "代码写谨慎一点"
    │
    ▼
Developer 判断：这是调参指令 → 降低自身温度参数
    │
    ▼
永久写入 AgentRole 配置（以后所有项目都生效）
    │
    ▼
私聊里静默确认（不通知群聊）
```

```
用户私聊 Developer: "写代码的时候参考一下 workspace 里的 auth.ts"
    │
    ▼
Developer 判断：这是指令注入 → 后续任务的提示词中加入参考要求
    │
    ▼
永久写入 AgentRole 配置
```

**判断逻辑由智能体自主决定**：用户的话是调参、追加提示词、还是普通闲聊。

### 4.7 配置变更通知规则

| 变更来源 | 通知方式 |
|---------|---------|
| **群聊中**用户说的 | 智能体改完配置后，在群里发一条通知 |
| **私聊中**用户说的 | 智能体改完配置后，**不通知群聊**（私聊保持私密） |

### 4.8 会话管理

- **一个智能体一个私聊窗口** — 用户和同一个智能体始终是同一个对话，持续累积上下文
- **会话持久化** — 历史消息存数据库，可回看
- **无路由、无帧栈** — 纯 1v1 对话，不触发群聊调度

### 4.9 工具使用

- 智能体已配置的工具在私聊中**可以直接使用**
- 没有配置所需工具时，**正常回复用户**（不触发能力协商，那是群聊里的事）

### 4.10 与 TestChat 的区别

| 维度 | TestChat | 1v1 私聊 |
|------|----------|---------|
| **持久化** | 瞬态（5 分钟清理） | 持久化（存数据库） |
| **项目上下文** | 无 | 可选（默认带入来源项目） |
| **历史记录** | 无 | 有（持续累积） |
| **Override** | 支持（临时覆盖） | 不支持（直接改配置，永久生效） |
| **调参/指令注入** | 不支持 | 支持（核心能力） |
| **工具使用** | 支持 | 支持（使用已配置工具） |
| **用途** | 快速测试提示词效果 | 正式对话 + 影响智能体行为 |
| **前端入口** | 智能体卡片弹窗 | 独立页面 |
| **每角色数量** | 可同时多个 | 每个智能体只有一个 |

## 场景对比矩阵

| **智能体数量** | 1 | 多个 | 多个 | 1 |

| **持久化** | 瞬态 | 持久化 | 持久化 | 持久化 |
